Vector Stores: Specialized Storage for Embeddings

Vector stores are specialized databases designed to store, manage, and efficiently search vector embeddings. Once we have OpenAI embeddings (1536-dimensional vectors), we need a purpose-built system to store them.

Why Vector Stores?

Unlike regular databases, vector stores solve fundamental challenges:

  • Scale: Applications often need millions of embeddings
  • Dimensionality: Each embedding has hundreds/thousands of dimensions
  • Search Performance: Finding similar vectors is computationally intensive
  • Associated Data: Maintaining connections between vectors and source documents

Example Document Data Structure

# What needs efficient storage in a vector store
document_data = {
    "id": "doc_42",
    "text": "LangChain is a framework for developing applications "
            "powered by language models.",
    "embedding": [0.123, -0.456, 0.789, ...],  # 1536 dimensions
    "metadata": {
        "source": "documentation.pdf",
        "page": 7,
        "created_at": "2023-06-15"
    }
}

Core Components

Vector stores combine two essential components:

  1. Vector Storage: The actual database persisting vectors and metadata
  2. Vector Index: Specialized data structure enabling efficient similarity search
⚠️ The Curse of Dimensionality:
As vector dimensions increase, computing similarities becomes increasingly expensive, requiring O(dN) operations for d dimensions and N vectors. This makes naive similarity search impractical for large-scale applications.

Search Patterns in Vector Stores

Modern vector stores support several search patterns:

1. Exact Search

  • Returns precise nearest neighbors
  • Computationally prohibitive with large collections
  • Best for small datasets or baseline testing

2. Approximate Search

  • Trades accuracy for speed using LSH, HNSW, or quantization
  • Measured by recall (% of true nearest neighbors retrieved)
  • Essential for production RAG systems

3. Hybrid Search

  • Combines vector similarity with text-based search (BM25)
  • Single query for semantic + keyword matching
  • Best of both worlds approach

4. Filtered Vector Search

  • Applies metadata constraints alongside vector similarity
  • Example: "Find similar documents from last 30 days"

Similarity Measures

Vector stores offer multiple similarity measures:

  • Inner Product: Useful for comparing semantic directions
  • Cosine Similarity: Normalizes for vector magnitude
  • Euclidean Distance: Measures L2 distance in vector space
  • Hamming Distance: For binary vector representations

Vector Indexing Strategies

Vector indexing solves the performance challenge of finding similar vectors without comparing against every single vector in the database (brute force), which is computationally prohibitive for even medium-sized data volumes.

Why Indexing Matters:
Without proper indexing, searching through vectors would be like finding a book in a library with no organization system – you'd have to check every single book. Proper indexing transforms vector search from O(n) to O(log n) complexity.

Common Indexing Approaches

  1. Tree-Based Structures: Hierarchically divide the vector space
  2. Graph-Based Methods: HNSW creates navigable networks of connected vectors
  3. Hashing Techniques: Map similar vectors to same "buckets"

Indexing Strategy Trade-offs

Strategy Search Complexity Memory Best For
Exact Search O(DN) Low Small datasets, 100% recall needed
HNSW O(log N) High Production systems, large-scale
LSH O(N^ρ) Medium Streaming data, frequent updates
IVF O(DN/k) Low Limited memory, balance needed
Product Quantization Varies Very Low Memory-constrained, massive datasets

Detailed Strategy Comparison

1. Exact Search (Brute Force)

How it works: Compares query vector with every vector in database

  • Pros: Easiest to implement, 100% recall, good baseline
  • Cons: Slow for large datasets, O(DN) complexity
  • Use when: Dataset < 100K vectors, testing/baseline

2. HNSW (Hierarchical Navigable Small World)

How it works: Creates layered graph with decreasing connectivity from bottom to top

  • Pros: Industry standard, O(log N) search, high accuracy
  • Cons: High memory usage, requires tuning M (connections) and ef (search depth)
  • Use when: Production systems, millions of vectors, high accuracy needed

3. LSH (Locality Sensitive Hashing)

How it works: Hash functions map similar vectors to same buckets

  • Pros: Good for dynamic data, tunable accuracy vs speed
  • Cons: Requires multiple hash tables
  • Use when: Frequent updates, streaming data

4. IVF (Inverted File Index)

How it works: Clusters vectors and searches within relevant clusters

  • Pros: Low memory, simple implementation, often combined with other methods
  • Cons: Requires cluster training (k-means)
  • Use when: Limited memory, need balance of speed/accuracy

5. Product Quantization (PQ)

How it works: Compresses vectors by splitting into subspaces and quantizing

  • Pros: Very low memory (10-20x reduction), good for massive datasets
  • Cons: Complex implementation, requires training codebooks
  • Use when: Memory-constrained, billions of documents, often with IVF

Practical Performance Example

import numpy as np
import faiss
import time

# Create sample data - 10,000 vectors with 128 dimensions
dimension = 128
num_vectors = 10000
vectors = np.random.random((num_vectors, dimension)).astype('float32')
query = np.random.random((1, dimension)).astype('float32')

# Exact search index
exact_index = faiss.IndexFlatL2(dimension)
exact_index.add(vectors)

# HNSW index (approximate but faster)
hnsw_index = faiss.IndexHNSWFlat(dimension, 32)  # 32 connections
hnsw_index.add(vectors)

# Compare search times
start_time = time.time()
exact_D, exact_I = exact_index.search(query, k=10)
exact_time = time.time() - start_time

start_time = time.time()
hnsw_D, hnsw_I = hnsw_index.search(query, k=10)
hnsw_time = time.time() - start_time

# Calculate overlap
overlap = len(set(exact_I[0]).intersection(set(hnsw_I[0])))
overlap_percentage = overlap * 100 / 10

print(f"Exact search time: {exact_time:.6f} seconds")
print(f"HNSW search time: {hnsw_time:.6f} seconds")
print(f"Speed improvement: {exact_time/hnsw_time:.2f}x faster")
print(f"Result overlap: {overlap_percentage:.1f}%")

Typical Output:

Exact search time: 0.003210 seconds
HNSW search time: 0.000412 seconds
Speed improvement: 7.79x faster
Result overlap: 90.0%
⚠️ The Trade-off:
Exact search guarantees finding true nearest neighbors but takes longer. HNSW provides approximate results significantly faster. For small datasets (10K vectors), the difference is minimal, but as datasets grow to millions/billions, exact search becomes prohibitively expensive while HNSW maintains logarithmic scaling.

Choosing the Right Strategy

Decision Guide:
  1. Dataset Size < 100K: Exact search viable
  2. Memory Limited: Consider IVF+PQ for compression
  3. Frequent Updates: LSH supports efficient updates
  4. Need Ultra-Low Latency: HNSW typically fastest
  5. Production Systems: HNSW or IVF+HNSW combination

Popular Vector Libraries

  • FAISS (Meta): Efficient similarity search, supports PQ, LSH, HNSW, CPU/GPU
  • Annoy (Spotify): C++ library, random projection trees
  • hnswlib: C++ library for HNSW algorithm
  • nmslib: Supports various algorithms including HNSW, SW-graph, SPTAG
  • SPTAG (Microsoft): Distributed ANN, k-d tree and balanced k-means approaches

Breaking Down the RAG Pipeline

Think of the RAG pipeline as an assembly line in a library, where raw materials (documents) get transformed into a searchable knowledge base that can answer questions.

1. Document Processing – The Foundation

Document processing prepares books for the library:

  • Loading: Using appropriate document loaders (PDF, HTML, text, etc.)
  • Transformation: Converting to standard format
  • Splitting: Breaking into smaller, meaningful chunks

Example: Processing a textbook into chapter-sized or paragraph-sized chunks while preserving context in metadata.

2. Vector Indexing – Creating the Card Catalog

Making documents searchable:

  • Embedding model converts each chunk into a vector
  • Vectors organized in a data structure (vector store)
  • Maintains connections between vectors and original documents

3. Vector Stores – The Organized Shelves

Vector stores are like organized library shelves:

  • Store document vectors and original content
  • Provide efficient search methods
  • Offer organization methods (HNSW, IVF) balancing speed and accuracy

4. Retrieval – Finding the Right Books

When a question arrives:

  1. Question converted to vector using same embedding model
  2. Vector store finds documents with most similar vectors
  3. Retriever applies additional logic:
    • Removing duplicate information
    • Balancing relevance and diversity
    • Combining results from different methods

Basic RAG Implementation

from langchain.prompts import PromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser
from langchain_community.document_loaders import JSONLoader
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS

# 1. Load documents
loader = JSONLoader(
    file_path="knowledge_base.json",
    jq_schema=".[].content",
    text_content=True
)
documents = loader.load()

# 2. Convert to vectors
embedder = OpenAIEmbeddings()
embeddings = embedder.embed_documents(
    [doc.page_content for doc in documents]
)

# 3. Store in vector database
vector_db = FAISS.from_documents(documents, embedder)

# 4. Retrieve similar docs
query = "What are the effects of climate change?"
results = vector_db.similarity_search(query)
